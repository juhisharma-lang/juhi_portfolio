The Problem

The Agile Centre of Excellence at any mid-to-large organisation maintains process documentation for scrum masters, portfolio managers, and delivery leads. The problem isn't that the knowledge doesn't exist. It's that it lives across fragmented, partially outdated wiki pages and email announcements that rarely catch up with the actual changes.

I identified two distinct pain points: onboarding overload, where new team members in their first 30-90 days face the highest retrieval burden with no single reliable reference; and process change lag, where significant changes get communicated by email but the wiki updates slowly, creating a window where people are working from stale guidance.

The real bottleneck is that a small CoE team absorbs a high volume of repetitive questions that good documentation should handle. This reframes what the product is: not just a self-serve lookup tool, but a force multiplier for the CoE.

Why RAG, Not Fine-Tuning

The knowledge base is dynamic. Process changes happen multiple times per year. Email announcements introduce new information before wiki pages update. Fine-tuning requires retraining every time knowledge changes, which is expensive, slow, and operationally complex. RAG retrieves from a live knowledge base, so when a page is updated the bot reflects it immediately with no model changes.

V1 uses standard RAG: one query, one retrieval pass, one answer. Predictable, every step traceable, knowledge base bounded, faster to ship. V2 introduces agentic RAG, where an orchestration layer can plan multiple retrieval steps, detect conflicts between sources, and surface both versions rather than blending them into a misleading answer.

Key Design Decisions

1
Metadata in every response

Source page name, author, and last-modified date travel with every chunk through the pipeline. Users making process decisions need to verify the source and judge how current it is. It also creates accountability: page owners can see when their content is being cited.

2
Three-state confidence signal

High (answer fully grounded), Partial (related content found but incomplete, state this clearly), Unknown (no relevant match, do not call the LLM, return an escalation directly). A partial answer presented as complete is actively harmful. This is a trust mechanism, not just a technical feature.

3
Freshness flag at 12 months

A 90-day threshold would flag stable, valid content as stale and erode trust in the warning itself. 12 months catches genuinely overdue content while leaving stable pages untouched.

4
Semantic chunking at heading boundaries

Fixed token splits create chunks that mix unrelated topics. CoE wiki pages are structured with headings where each section covers one coherent topic. Splitting at headings means each chunk maps to how users actually ask questions.

5
Calibrated distance thresholds from real queries

Initial thresholds (below 0.8 = High) were wrong for all-MiniLM-L6-v2. The model operates in a higher distance range. A timesheet query returned a correct answer at 0.9366. Thresholds were recalibrated from actual results, not assumptions.

North Star and Measurement

The north star metric is time-to-independence: the point where a new team member can answer process questions without escalating to the CoE. Target is to reduce from the current 60-90 day window to under 30 days.

This metric is deliberately not measurable until V3, when user identity tracking and cohort analytics exist. V1 and V2 use proxy metrics: helpful answer rate, escalation rate, abandonment rate, regeneration rate, and latency at the 95th percentile. The V1 goal is to establish a baseline before claiming the before-and-after story.

Eval Results

A 15-question evaluation set was run before considering V1 complete. 12 of 15 passed. The 3 failures were documented as findings, not quietly dropped. One failure: multi-hop questions that required synthesising two separate wiki pages returned only partial answers. This is expected behaviour for standard RAG and is the primary motivation for the V2 agentic layer.

What I Learned Building This

The product principle that governed every decision: speed without trustworthiness is dangerous. A fast answer from a degraded wiki is worse than no answer. It creates false confidence. This shaped the confidence signal, the freshness flag, the scope boundary, and the escalation path.

The threshold calibration lesson was the most concrete: never assume your embedding model behaves like the default examples. Run real queries, observe actual distances, then set your thresholds. The first pass was wrong. The second pass, grounded in observed data, was correct.

The hardest product decision was keeping clarifying questions out of V1. They would have made the demo look more capable. But adding orchestration complexity before retrieval is proven creates a system that fails in ways that are hard to debug. V1 is standard RAG. V2 earns the complexity.

Roadmap

V1 Built

Standard RAG. Natural language Q&A, source citation, confidence signalling, freshness flag, escalation path.

V2 Designed

Agentic RAG. Multi-source retrieval, conflict detection, clarifying questions, repeat query digest to CoE.

V3 Strategic

Wiki gap detection, auto-draft suggestions, usage analytics, guided onboarding paths, JIRA integration.
