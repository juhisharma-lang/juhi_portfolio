The Original Hypothesis

A reasonable assumption that research killed cleanly.

The starting point was a reasonable assumption. Working professionals and students preparing for career milestones carve out time to learn. When something urgent comes up, that learning time is the first to disappear. The frustration, the assumption went, came from not knowing how to reprioritize. So the original plan was a GenAI assistant that would reason through those tradeoffs.

20 survey responses, 3 in-depth interviews, a behavioural science review, and a competitive scan later: research killed that hypothesis cleanly.

What Research Found

Not a prioritisation problem. A re-entry problem.

People were not struggling with the "what to prioritize" question. They were struggling with the "how do I start again" question. The pattern was almost identical across respondents: miss a day, feel guilty, avoid the app, lose a week, eventually attempt to restart with emotional overhead attached.

Signal 01
Collapse, not disruption
When a disrupted day hit, learning collapsed for days or a week, not just one day.
Signal 02
Emotional pattern
Frustration, guilt, overwhelm. The feeling of having to start over from zero, even when objectively they were not.
Signal 03
What they wanted
Respondents pointed toward nudges and structure, not AI reasoning. Something that said: your day broke, here is the smallest useful step.

Behavioural science reinforced the findings. Unfinished tasks stay mentally active after the day ends, creating cognitive drag. Forming a specific if-then plan significantly reduces that mental interference. The science was describing exactly the re-entry problem respondents were living.

The Pivot

From decision-support tool to re-entry support system.

The GenAI reprioritisation concept was dropped. The new core question was not "what should you do today?" It was: when you fell off, how fast did you come back?

That reframe sounds small. It changed every design decision that followed. Completion rate was the obvious north star for a habit app. The problem is that it is binary and punitive. A user who missed three days but returned the next day is doing better, in the sense that matters, than a user who has not missed yet but is slowly losing engagement. Completion rate cannot tell those two users apart. Recovery speed can.

"Five products reviewed. All of them track completion. None of them track what happens after you do not complete."

What Changed in V2

A focus shift, not a rebuild.

V1 was Telegram-first and generic. Expert review surfaced four clear pivots: remove the Telegram dependency, add journey structure, make recovery speed the visible metric, and build a miss recovery UX. V2 delivered all four.

The sharpened persona: learners working toward structured milestones — certifications, career transitions, skill-building programs — where the cost of falling off is higher and the re-entry problem is more acute. The milestone creates urgency, which is the product's leverage. The 4-state model holds. Recovery speed as north star holds. What changed is the journey templates, the plant-based emotional feedback layer, and background push notifications designed around how learners actually structure their goals.

The Model

Four states. One pure function. No streaks.

The product is not built around a streak. It is built around four states, each of which calls for a different response from the system.

A
On Track
Checking in consistently. Dashboard is calm and affirming. No friction introduced.
B
Missed Yesterday, No Plan
Highest-risk state. Dashboard surfaces a low-friction restart prompt, not a reminder of what was missed. No guilt.
C
Restart Committed
User has set a specific return date and time. The system holds the intention so the user does not have to carry it mentally.
D
Deliberate Pause
User has proactively flagged a busy period. This is not a failure state. Treated as a planned interruption. Re-entry managed when the pause expires.

All four states are driven by a single pure function, getDashboardState(), which determines message, prompt, and tone at any given moment. Grounded in implementation intention research.

What Was Built

Mobile-first. Deployed. Beta-ready.

Learners Buddy is a web-based mobile-first app. Stack: Vite, React, TypeScript, Tailwind CSS, shadcn/ui. No native app — an intentional decision to test core behaviour without app store friction or distribution overhead.

Core features: 10 pre-built journey templates plus a custom builder; daily check-in under 60 seconds; plant visual that grows on sessions, wilts on 2+ consecutive misses, and recovers the moment a session is logged; dashboard with recovery speed as the hero metric; miss recovery ladder with escalating copy (banner → recovery sheet → feedback mode at 7+ misses); editable weekly goals with push-to-next-week and replace flows; pause flow that suspends miss streak counting.

Backend is live. Supabase mirrors all user state in real time — every check-in, pause, and goal edit is persisted to the cloud. Push notifications fire even when the browser is closed, delivered via Supabase Edge Function on a 30-minute cron schedule using Web Push (VAPID).

Metrics

The north star maps to what the user feels.

Why Not Completion
Binary and punitive. Cannot distinguish a user recovering well from one slowly disengaging.
Phase 1 Signal

Phase 2 Goal

The Judgment Call

Killing something you already designed.

The most interesting PM moment in this project is not the pivot itself. It is the discipline the pivot required. By the time the research came back pointing away from the original hypothesis, the GenAI reprioritisation concept had already been designed. There was investment in the idea. Killing it was uncomfortable.

"Treating research as a genuine input that can and should change direction, not as validation material for a decision already made, is the most transferable thing this project demonstrates."
